Specification constants

These two 128-bit constants are defined in the HIS SHE specification. They are used as plaintext inputs to AES-ECB with the MASTER_ECU_KEY to derive K1 (encryption key) and K2 (MAC key) for every key update operation.

KEY_UPDATE_ENC_C
0x010153484500800000000000000000B0
Constant for deriving K1 (encryption key). Byte 0x01=version, 0x53484500="SHE\0", 0x80=separator, 0xB0=purpose flag.
KEY_UPDATE_MAC_C
0x010253484500800000000000000000B0
Constant for deriving K2 (MAC key). Identical to ENC_C except byte[1] = 0x02 distinguishes MAC derivation from encryption derivation.
K1 = AES_ECB_Encrypt(MASTER_ECU_KEY, KEY_UPDATE_ENC_C)
K2 = AES_ECB_Encrypt(MASTER_ECU_KEY, KEY_UPDATE_MAC_C)

M1 — ECU identity block
M1 = UID[120 bits] | ID[4 bits] | AuthID[4 bits] = 128 bits / 16 bytes
Identifies which ECU (UID), which slot to update (ID), and which key authenticates this update (AuthID). Binds the update to a specific ECU — prevents cross-ECU injection attacks.
UID (120 bits)
ID (4 bits)

M2 — Encrypted new key
plaintext: Counter[28] | FID[4] | 0x00*12[96] | NewKey[128] = 256 bits
M2 = AES_CBC_Encrypt(K1, IV=0, plaintext) = 256 bits / 32 bytes
NewKey is never transmitted in plaintext. It is encrypted using K1 (derived from MASTER_ECU_KEY). Counter prevents replay. FID sets key access flags. Padding fills to 256-bit block boundary.
Counter (28 bits)
FID (4 bits)
Padding (96 bits)
NewKey (128 bits)
M3 — Authentication MAC
M3 = AES_CMAC(K2, M1 || M2) = 128 bits / 16 bytes
SHE verifies M3 FIRST before doing anything else. If MAC check fails, the entire update is rejected. M3 covers M1+M2 together — so any tampering with identity, counter, flags, or encrypted key is detected immediately.
M4 + M5 — Proof of update (SHE response)
M4* = UID[120] | ID[4] | AuthID[4] | Counter+1[28] | 1[1] | 0..0[99] = 256 bits
M4 = AES_ECB_Encrypt(NewKey, M4*[0..127]) || AES_ECB_Encrypt(NewKey, M4*[128..255])
M5 = AES_CMAC(K2, M4) = 128 bits / 16 bytes
M4 and M5 are generated BY the SHE hardware AFTER successful key storage. The key server verifies M4+M5 to confirm the ECU actually stored the new key. Cannot be faked without knowing NewKey. The proof bit (1) and Counter+1 confirm correct storage.
M4 — encrypted proof 32 B
M5 — MAC over M4 16 B

Verify M4 M5 — key server check
After sending CMD_LOAD_KEY, the ECU returns M4 and M5. The key server must verify:
1. Decrypt M4 with NewKey → recover M4* → check UID, ID, AuthID, Counter+1
2. Recompute CMAC(K2, M4) → must match M5
If both checks pass → key was correctly stored in SHE hardware.
